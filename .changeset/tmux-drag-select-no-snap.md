Mouse copies in tmux panes now reach the system clipboard, and drag-select no longer traps you in copy-mode. Because kobe forces `mouse on` (so the Tasks pane click-to-switch and the Ops file tree's click/scroll work), mouse selection runs through tmux's copy-mode rather than the terminal's native selection. Two fixes: (1) `set-clipboard on` makes copy-mode copies emit OSC 52, so a dragged selection lands on the system clipboard — `Ctrl/Cmd+V` in another app now works, not just tmux's internal paste buffer. (2) `MouseDragEnd1Pane` is pinned to tmux's default `copy-selection-and-cancel` (copy, then exit copy-mode back to live output). An interim build had bound it to a stay-in-copy-mode action to avoid the view snapping to the bottom, but staying in copy-mode trapped you in the modal scrollback viewer — typing (especially multi-byte IME input) went to copy-mode instead of the engine and the screen stopped refreshing until you pressed Ctrl+C. That trade-off is inherent to tmux: not snapping requires staying in copy-mode, which blocks input. To select without snapping the view, use the terminal's native bypass — hold Shift while dragging (Option on macOS Terminal.app) — which skips tmux's mouse handling entirely and selects straight to the system clipboard.

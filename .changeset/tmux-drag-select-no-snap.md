Mouse text selection now behaves like a normal GUI terminal: releasing the drag leaves the text highlighted, the pane no longer snaps to the live bottom, and the highlight stops growing once you let go. Because kobe forces `mouse on` (so the Tasks pane click-to-switch and the Ops file tree's click/scroll work), drag-selecting hit tmux's default `MouseDragEnd1Pane` binding, which runs `copy-selection-and-cancel` — `-and-cancel` exits copy-mode, which both dropped the highlight and jumped the view to the bottom, so copying scrollback yanked you away from what you were reading. The drag-end binding (in both the `copy-mode` and `copy-mode-vi` tables) now runs `copy-selection-no-clear` (copy into the buffer, keep the highlight, stay at the current scroll position) followed by `stop-selection` (freeze it so a later wheel-scroll or mouse move no longer extends the selection). Drag again to start a fresh selection, or press `q` to return to the live bottom. Tip: to get fully native selection straight to the system clipboard (no copy-mode at all), hold Shift while dragging — or Option on macOS Terminal.app — which bypasses tmux's mouse handling entirely.
